APolGe: Die Befehlsverarbeitung (name.tfossi.apolge.common.cmd) Technical Referenz

UP


Vorgesehen für alle Klassen und Schnittstellen der Befehlssteuerung.

Die Commandklassen steuern die Befehlsabwicklung. Der Einsprung ist immer gleich 
public void command()
Der ConcreteCommand kennt den oder auch mehrere die Methoden des Receivers und ruft diese ggfs. entsprechend der Parameter auf. Die eigentliche Befehlsausführung (Code) findet also im Receiver statt.
Es gibt ein paar Standardparameter, die immer angeboten werden:

public void back();
public void contentRecorder();
public void recRecorder(String macroname, String comment);
public void stopRecorder();
public void playRecorder(String ... value);
public void exit();
public void credit();

Beschreibung Command-Pattern

Die Befehlssteuerung ist nach dem Command-Pattern designed worden.
  • Command: abstract class ACmd
      ist die zentrale Klasse des Commandpatterns. Hier wird der Receiver eingetragen, die Befehlsparameter, binäre Daten, Informationen für das SWT und weitere Screeninfos.
  • Command: interface ICmd
      ist die Schnittstelle zur Ausführung der Befehle und der Vorbereitung. Für das SWT stehen noch besondere Methoden bereit.
  • Command: interface IBuildSWTWidget
      spezielle Schnittstelle für SWT-Steuerung
  • Command: class Command
      dient zur Parameterauswertung
  • Command: class ConcreteCommand
      enthält den eigentlichen Befhelfsaufruf
  • Receiver: class AMenuReceiver
      Alle Methoden zur Befehlssteuerung
  • Receiver: interface IGeneralReceiver
      Enthält die Signaturen der Methoden, die der Receiver beherrschen muss
  • Receiver: interface IReceiver
      Marker
  • Receiver: interface INetReceiver
      Enthält die Signaturen der Methoden für die Netzsteuerung
  • Receiver: class ConcreteMenu
      Die konkrete Klasse, die die Commands empfängt und verarbeitet.

Beschreibung der Anmeldung von Commands

Jedes Command muss instanziert und angemeldet werden, sonst kann es nicht ausgeführt werden. Dazu wird ein Script genutzt, was es ermöglicht, nur mit Teilcompilierung des Codes neue Befehle einzubauen und einfach Befehlszuordnungen abzuschalten.

Die Instanzierung erfolgt im Konstruktor der Klasse CommandMaps(). Aus der Konfigurationstabelle config wird das Feld CMP ausgelesen und die Commands per Reflection instanziert.

Es gibt eine weitere Liste im Konfigurationsscript config, in der beschrieben ist, welche Befehle den Modellen (MENU) oder den Views (..VIEW) zugewiesen sind.

Die Klasse new CommandMaps().fetchList(..VIEW) ordnet dann (hier: dem View) die gültigen Befehle zu.

Ein weitere Liste (GRAYED...) im Konfigurationsscript config definiert die initial ausgegrauten Menüs.
